## Codebase Patterns
- ChatMessage is discriminated union on `role` field — narrow before accessing role-specific fields
- `TextChatMessage` has `content`; `ToolChatMessage` has `toolId`, `tree`, `status`
- UITree type from `@cloudflare/kumo/streaming` (imported at line 92)
- Action system: ActionEvent → ActionDispatch → handlers (action-types.ts, action-registry.ts)
- Streaming pipeline: readSSEStream → createJsonlParser → useUITree → UITreeRenderer
- Pre-existing typecheck errors in shiki/echarts modules — ignore
- `streamPanelB` sends history to API — must filter to TextChatMessage only
- Chat message rendering at ~line 2170, must handle all 3 roles explicitly
- JSONL generator pattern: build `readonly object[]` of RFC 6902 ops, `JSON.stringify` each, `.join("\n")`; feed to parser with trailing `\n` or use `flush()`
- Custom action handlers: define `ActionHandler` functions → `ActionHandlerMap` → `createHandlerMap(custom)` merges with BUILTIN_HANDLERS; use merged map in `dispatchAction`
- Tool card rendering: `UITreeRenderer` with `onAction` prop directly in chat loop; no wrapper bubble chrome; `w-full overflow-hidden` prevents sidebar overflow
- Tool action dispatch: `handleToolAction` finds tool message by `event.params.toolId`, passes its `tree` to `dispatchAction`; uses `messagesRef` to avoid stale closure
- Status-based tool card styling: CSS wrapper with `cn()` around UITreeRenderer; `opacity-50 pointer-events-none` for disabled; `onAction={undefined}` to suppress dispatch; Text heading variants render as `<h1>`/`<h2>`/`<h3>` not `<span>`
- Tool middleware intercept: early return in `handleSubmit` before panel streams; match message → stream from `/api/chat` with `TOOL_CONFIRMATION_PROMPT` → `createJsonlParser` + `applyPatch` → update tool message tree incrementally; separate `toolAbortRef`
- `skipSystemPrompt: true` + `systemPromptOverride` on `/api/chat` gives focused LLM task without polluting main prompt
- Tool action routing: intercept `event.actionName` (`tool_cancel`/`tool_approve`) in `handleToolAction` before `dispatchAction`; use `updateToolMessageStatus(toolId, newStatus)` for immutable state transitions
