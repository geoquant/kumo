# Progress Log
PRD: kumo-catalog-runtime-advantage
Started: 2026-03-09

## Codebase Patterns
- Registry codegen is the SSOT: add new metadata in `packages/kumo/scripts/component-registry/` and regenerate committed artifacts instead of hand-editing files in `packages/kumo/ai/` or `packages/kumo/src/generative/`.
- Generative classification lives in `packages/kumo/scripts/component-registry/generative-map-generator.ts`; downstream tests treat the generated `component-manifest.ts` as the contract for prompt-builder, validator, and component-map behavior.
- Behavior-only runtime hints should come from one small overlay in `packages/kumo/scripts/component-registry/metadata.ts`, not scattered per-consumer allowlists.
- When a generated artifact is string-compared in tests, rerun the generator after formatting so checked-in output matches the generator's exact serialization.
- Legacy UITree compatibility should normalize `parentKey -> children` before adapting, then use generated behavior metadata to decide which dynamic props become `$bind` values and which legacy `action` handlers map to `change` vs `press` events.
- Async import tests must await the import promise directly; wrapping `await import(...)` inside `expect(async () => {}).not.toThrow()` can leave Vite module fetches pending and make Vitest fail with unhandled worker RPC shutdown errors.
- Freshness checks should validate both layers of the pipeline: demo-source -> `dist/demo-metadata.json`, then metadata + component source -> all checked-in registry artifacts.
- Loadable can stay host-first externally while using AppSpec internally by mirroring each container's legacy UITree into a canonical spec and treating `/state`/`/meta` patches as spec-native updates.
