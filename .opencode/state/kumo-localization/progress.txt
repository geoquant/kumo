# Progress Log
PRD: kumo-localization
Started: 2026-03-05

## Codebase Patterns
- New modules go in `packages/kumo/src/localize/` dir
- Interface pattern: `readonly` fields for immutability, grouped by section with doc comments
- Parameterized translation keys typed as functions `(arg: type) => string` (no MessageFormat runtime)
- `$`-prefixed fields for metadata ($code, $name, $dir) to avoid collision with translation keys
- Pre-existing issues: docs astro typecheck has 1 error (SearchDialog), kumo tests have 70 unhandled rejections in deep-imports.test.ts (Vite worker RPC closing) — both pre-existing
- `useSyncExternalStore` pattern for DOM attribute reactivity: lazy observer in `subscribe`, snapshot reads from DOM, server snapshot returns static default
- Files with JSX must use `.tsx` extension (even non-component modules like `index.tsx` in localize/)

## Task - core-2
- Created `packages/kumo/src/localize/registry.ts` with `registerTranslation()` and `getTranslation()`
- `registerTranslation()` accepts variadic `KumoTranslation` args, stores in Map keyed by `$code`
- `getTranslation(lang)` resolves: exact match → language prefix (before first hyphen) → first registered fallback
- First registered translation tracked via `fallbackCode` variable
- Multiple calls merge into existing registry (Map.set doesn't clear)
- `_resetRegistry()` exported for test cleanup
- Files changed: `packages/kumo/src/localize/registry.ts` (new)
- **Learnings:** Keep module-level mutable state minimal (one Map + one string); `_resetRegistry` with underscore prefix convention for test-only exports

## Task - core-3
- Created `packages/kumo/src/localize/use-locale.ts` with `useLocale()` hook
- Uses `useSyncExternalStore` with three functions: `subscribe`, `getSnapshot`, `getServerSnapshot`
- `MutationObserver` created lazily on first subscription, disconnected when last subscriber removed
- Subscriber tracking via `Set<() => void>` — observer only exists while subscribers are active
- Fallback chain: `document.documentElement.lang` → `navigator.language` → `"en"`
- Server snapshot always returns `"en"` for SSR
- Files changed: `packages/kumo/src/localize/use-locale.ts` (new)
- **Learnings:** `useSyncExternalStore` is the right pattern for DOM attribute reactivity — avoids useEffect/useState dance; lazy observer creation prevents work in module scope (important for SSR)

## Task - core-4
- Created `packages/kumo/src/localize/index.tsx` with `KumoLocaleProvider` and `useLocalize()`
- `KumoLocaleProvider` wraps a context provider; optional `locale` prop overrides detection for subtree
- `useLocalize()` returns `{ term, date, number, lang, dir }` memoized on resolved locale
- Type-safe `term()` via `TermFn` interface with two overload signatures: `SimpleKey` (string values) and `ParameterizedKey` (function values with inferred args)
- `SimpleKey`/`ParameterizedKey` computed via mapped conditional types over `KumoTranslation`
- `date()` and `number()` delegate to `Intl.DateTimeFormat`/`Intl.NumberFormat` with resolved locale
- Locale resolution: `KumoLocaleProvider` override → `useLocale()` (html lang → navigator → 'en')
- Re-exports `registerTranslation` and `KumoTranslation` type for single-file public API
- File is `.tsx` (not `.ts`) because it contains JSX for the context Provider
- Files changed: `packages/kumo/src/localize/index.tsx` (new)
- **Learnings:** JSX in localize module requires `.tsx` extension; `useMemo` on resolved locale string gives stable identity for the returned object; `TermFn` overloads give type safety without runtime cost
