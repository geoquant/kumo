# Progress Log
PRD: playground-iteration-inspection-parity
Started: 2026-03-08

## Codebase Patterns
- Verify the current branch is `geoquant/streaming-ui` before starting any new work for this PRD; do not branch away for playground parity tasks.
- Keep shared playground state shapes in `packages/kumo-docs-astro/src/lib/playground/types.ts` and pure state transitions in `packages/kumo-docs-astro/src/lib/playground/state.ts`.
- Migrate mirrored panel UI state first (tabs, status, raw JSONL, action logs, layout flags) and leave stream/runtime hooks intact until later tasks need deeper consolidation.
- For inspection tabs, derive a memoized JSON-friendly view model in `src/lib/playground/` and render it through `HighlightedCode` with the same copy-button pattern used by JSONL and TSX tabs.
- When an editor can override streamed output, keep the streamed tree and local override separate; render `override ?? streamTree` and only patch the override locally when interactive actions fire.
- Keep catalog reference data UI-agnostic: normalize registry components and action definitions in `src/lib/playground/catalog-data.ts`, then let later UI layers consume sorted arrays instead of raw registry maps.
- For shared explorers, anchor open/close state in the top-level layout reducer and render one responsive overlay that becomes a desktop sheet and a mobile full-screen view instead of duplicating separate desktop/mobile components.
- For TSX export, keep the internal panel tab value stable (`code`) but change the visible label and pass explicit export component names from the panel boundary so generated modules stay panel-specific.
- For persisted splitter layouts, use `react-resizable-panels` `Group` + `Panel` + `Separator` with stable group/panel ids and `useDefaultLayout()` so localStorage restore works without custom storage plumbing.
- For breakpoint-specific shells, mount only one shell at a time using a viewport flag; reuse shared `PanelContent`, `PanelHeader`, and catalog content instead of rendering hidden desktop/mobile copies in parallel.
- For docs playground tests, keep most coverage in pure `src/lib/playground/*.test.ts` modules, then add one focused `PlaygroundPage` smoke suite with mocked transport/UI wrappers to verify cross-cutting flows without needing full browser automation.
